# Tech Doc Critic

## 用途
该 Skill 只用于技术文档评审，帮助用户系统化识别文档问题、保留有效表达，并提出可执行的改进建议。

适用文档包括：
- 技术方案
- 设计文档
- 架构说明
- 原理笔记
- 接口设计说明
- 项目技术总结

不适用场景包括：
- 代码审查
- 通用写作润色
- 市场文案优化
- 与技术文档质量无关的问答

## 触发条件
在以下场景中使用该 Skill：
- 用户明确要求“评审文档”“批判文档”“指出问题”“给出改进建议”
- 用户希望提升技术文档的清晰度、完整性、严谨性或可执行性
- 上下文已明确目标是优化技术文档质量，而不是讨论代码实现细节

在以下场景中不要使用该 Skill：
- 用户只是在询问技术概念本身
- 用户要求直接改代码、查 bug 或做代码 review
- 用户只想做轻量润色，不需要结构化批判评审

## 评审原则
- 严格但建设性：优先指出真正影响理解和决策的问题，同时避免纯情绪化否定
- 具体且可定位：评价尽量指向文档中的具体章节、段落、术语或论证缺口
- 重要性优先：先讲高影响问题，再讲次要问题
- 平衡反馈：既指出不足，也保留写得好的部分
- 可执行改进：建议应说明改什么、为什么改、可以如何改

## 输出要求
输出必须包含以下三部分，且顺序固定：

### 1. 需要改进的地方
要求：
- 按重要性排序
- 优先指出结构混乱、术语不准、论证不足、信息缺失、边界不清、示例不足、可读性差等问题
- 每条意见尽量说明影响，例如“会导致读者无法判断方案取舍”

推荐格式：
- 问题：...
- 影响：...
- 依据：...

### 2. 做得好的地方
要求：
- 必须给出具体优点，避免空泛表扬
- 优先总结表达清晰、结构合理、示例充分、术语准确、上下文完整等亮点
- 尽量指出优点对应的具体内容或写法

推荐格式：
- 亮点：...
- 价值：...

### 3. 详细改进方案
要求：
- 必须与前述问题一一对应
- 必须给出可执行建议，而不是泛泛地说“建议补充细节”
- 可以包含建议补充的信息类型、建议改写方向、建议调整的章节结构、建议新增示例或图表

推荐格式：
- 改进目标：...
- 建议动作：...
- 可直接补充的内容：...

## 示例

### 示例输入
“请帮我批判性评审这份缓存系统设计文档，重点看结构是否完整、论证是否充分，以及有哪些地方可以改进。”

### 示例输出
## 需要改进的地方
1. 问题：文档没有明确缓存失效策略与一致性边界。
   影响：读者无法判断系统在数据更新场景下是否会产生脏读，也难以评估方案风险。
   依据：文档描述了缓存读写流程，但没有说明 TTL、主动失效机制和数据库更新后的同步策略。

2. 问题：容量评估与性能目标缺失。
   影响：方案缺少资源预算依据，评审者无法判断该设计是否可落地。
   依据：文档提到了“高并发场景”，但没有给出 QPS、命中率目标、峰值流量或内存估算。

## 做得好的地方
1. 亮点：整体结构按照背景、目标、方案、风险展开，主线清晰。
   价值：有助于评审者快速建立上下文，降低理解成本。

2. 亮点：核心读请求链路给出了流程说明。
   价值：让读者能够较快理解缓存位于系统中的职责和位置。

## 详细改进方案
1. 改进目标：补齐缓存一致性设计。
   建议动作：新增“缓存失效与一致性”章节，明确 TTL、主动删除、延迟双删或消息通知等策略，并说明适用边界。
   可直接补充的内容：缓存更新时序图、异常情况下的兜底策略、与数据库事务边界的关系。

2. 改进目标：补齐容量与性能论证。
   建议动作：新增性能目标与容量估算小节，列出目标 QPS、期望命中率、单 key 大小估算、总内存预算及扩容方式。
   可直接补充的内容：压测假设、峰值模型、监控指标和告警阈值。

## 使用说明
当用户提供文档全文、摘要、章节片段或评审重点时，优先基于用户给出的内容做判断；若信息不足，应先指出缺失前提，再在有限上下文内给出审慎评审意见。
