在连续两个月使用codex & cursor 后，核心体感是：Coding Agents 可以显著提升实现速度，但它不能自动带来工程质量。它更像是一个执行力极强、上下文记忆有限、风格会随提示词漂移的工程伙伴。如果我们只把它当成“自动写代码机器”，短期非常爽，长期就会不断把隐含决策、局部最优和未经验证的实现沉淀进项目里。

这也是一次性项目和长期项目最大的区别。一次性项目的目标通常是快速验证想法，代码只要能跑、能展示、能证明方向即可；长期项目的目标则完全不同，它需要在后续需求变化、多人协作、线上问题排查和持续迭代中保持可理解、可修改、可验证。前者可以容忍大量临时决策，后者必须把决策显式化。

如果不 review coding-agents 的产物，agents每次过度设计，每次隐形选择，都会层层叠加，最后变成一个技术债。到最后你的选择很可能是：基于这个已有的屎山，新建一个干净分支/仓库，推倒重来；

Anyway，针对一些短期项目，随agents所欲非常爽。但本文想分享，针对需要交付的项目，如何和 coding-agents 协同开发稳定的代码。

0. **慢即是快，慢即是快，慢即是快。** 这里的慢不是降低效率，而是不要把“生成得快”误认为“交付得快”。长期项目里，真正耗时的往往不是写代码，而是返工、排查、补测试、清理边界和修复错误抽象。所以前期多花一点时间把需求、上下文、边界和验收标准讲清楚，反而是在减少后面的维护成本。

1. **Plan 阶段**：不要一上来就让 agents 生成一个 plan，更不要直接让它开写。先和 agents 讨论需求，让它复述它理解的目标、非目标、已有上下文和风险点；再让它去读相关文档、已有代码、AGENTS.md、README、测试用例或者历史实现。等它拥有足够上下文后，再产出一个可以分步执行的 plan。一个好的 plan 不应该只是任务列表，而应该说明每一步改哪里、为什么改、怎么验证、失败了如何回滚。

2. **Spec 阶段**：复杂需求一定要先把规格写出来，尤其是长期项目。Spec 里至少要包含：功能目标、明确不做什么、核心数据结构、关键 API、模块边界、异常状态、性能或体验要求、验收标准。它还要提前规定实现边界：哪些文件可以改、关键函数的 API 是什么、模块之间如何传递数据、哪些行为暂时不做、每一步如何验证。假设我们现在要做一个功能：摄像头识别手势并模拟播放鼓声，那么我们应该定义清楚：用户打开摄像头后，页面会在画面上叠加虚拟鼓区，识别双手位置、移动速度和敲击动作，把不同区域映射成军鼓、底鼓、镲片等音色，并尽量做到低延迟、少误触、可配置。这个项目后续还可能加校准流程、灵敏度调节、录制回放、节拍器、教程模式和不同鼓组音色，所以 Spec 不能只写“识别手势并播放鼓声”，而要写清楚摄像头权限失败怎么处理、手势识别输出什么结构、一次敲击如何去重、鼓区坐标是否可配置、音频是否预加载、延迟多少以内算可接受。

3. **Execution 阶段**：如果 Spec 写得足够清楚，Execution 阶段就不应该再讨论大的设计问题，而是严格按 Spec 执行。这个阶段最重要的是控制改动范围：一次只完成一个可验证的小任务，不顺手重构无关代码，不临时改变 API，不把未讨论过的抽象塞进实现里。每完成一步，都应该能对应到 Spec 里的某个目标和验收标准；如果实现过程中发现 Spec 不完整，就先停下来补 Spec，而不是让 agent 继续猜。

4. **Review 阶段**：目前agents已经不太可能存在语法问题，review 需要关注在：关键逻辑、模块边界、状态设计、异常处理、是否过度设计、是否绕过已有抽象、是否引入新的隐式耦合，实现并不完整的mock。

5. **Validation 阶段**：验证要尽量自动化，关注单元测试、端到端测试和可复现样例。

6. **Context 阶段**：把有价值的上下文沉淀进仓库，而不是散落在聊天记录里。需求、设计方案、约束、检查清单、验证命令、历史坑点，都应该和代码一起维护。但上下文不是越多越好：临时草稿、过期方案和已经被推翻的中间版本要及时清理，避免误导模型；重要的决策原因和取舍过程则可以沉淀成 ADR（Architecture Decision Record，架构决策记录）、设计记录或 changelog。稳定的项目规则适合放在 AGENTS.md、README 或 docs 里，重复执行的工作流才适合沉淀成 skill，例如“执行发布前检查”“正式发版流程”。Skill 不应该是文档垃圾桶，而应该包含明确触发场景、输入要求、执行步骤、输出格式和验证命令，比如 Anthropic 的 skills 仓库：https://github.com/anthropics/skills。

7. **Tooling 阶段**：这是额外的交付阶段，应该放在项目已经跑通、关键路径已经明确之后再做。它的目标不是一开始就把所有工具建设完，而是把交付过程中反复使用、已经证明有价值的验证动作 CLI 化。例如 `npm run validate`、`npm run test:gesture -- --fixture hit-snare.mp4`、`npm run test:audio`、`npm run latency:measure`。长期项目不能只靠 agent 打开页面、看结果、凭感觉判断功能是否正常，而应该在最后把这些稳定动作沉淀成命令入口。CLI 的价值在于输入明确、输出稳定、退出码可判断，也方便 agents 在后续迭代中反复运行、观察、修正。工具越结构化，agent 越不容易靠猜。


希望 agents 继续进化，未来可以自己维护 plan、spec、review、validation、context 这些流程，甚至主动发现项目里的上下文缺口和验证缺口。但在那一天到来之前，我们仍然需要先把这些流程显式化、结构化、工具化。
